# 📋 Roadmap: ESP32 Smart Bridge (RTOS Edition)

## Фаза 1: Инфраструктура (Core & Foundation)
*Цель: Создать надежный фундамент для RTOS-приложения.*

- [x] **1.1. Адаптация протокола** (выполнено ранее, требует переноса в новую структуру)
    - [x] `lib/protocol` (остается как legacy или интегрируется в Services)
    - [x] Анализ и портирование логики v1.2 в `Protocol_Srv`.
- [x] **1.2. Настройка проекта PlatformIO**
    - [x] Обновить `platformio.ini` (build_flags, src_filter).
    - [x] Создать структуру папок `App/...` в корне проекта.
- [ ] **1.3. Реализация App/Core (Vertical Slice Goal)**
    - [x] `SystemState.h/cpp`: Структура данных (HEALTH_DATA v1.2), Mutex, Singleton.
    - [x] Парсинг Big-Endian данных из пакетов UART (0x03, 0x04).
    - [ ] `EventGroups.h`: Определение битовых флагов (BIT_WIFI_CONNECTED).
    - [ ] `Config.h`: Глобальные константы и настройки WiFi.

## Фаза 2: Драйверы и Сервисы (Drivers & Logic)
*Цель: Реализовать работу с железом и бизнес-логику без привязки к задачам.*

- [ ] **2.1. App/Drivers**
    - [x] `Uart_Drv`: Инициализация Serial2, чтение/запись.
    - [x] `Fifo_Drv`: ООП-реализация кольцевого буфера на базе `lib/protocol`.
    - [x] Интеграция `Fifo_Drv` в `Uart_Drv` для фонового приема данных.
    - [ ] `WiFi_Drv`: Базовая логика подключения (без веб-сервера).
- [ ] **2.2. App/Services**
    - [ ] `Logger_Srv`: Потокобезопасный логгер.
    - [ ] `Cmd_Srv`: Очередь команд (xQueue wrapper).
    - [x] `Protocol_Srv`: Интеграция парсера v1.2 (из lib/protocol).
    - [ ] `Api_Srv`: Генерация JSON status из данных `SystemState`.

## Фаза 3: Задачи RTOS (Execution)
*Цель: Оживить систему, запустив параллельные потоки.*

- [ ] **3.1. Uart_Task (Core 1)**
    - [x] Цикл приема данных, обновление `SystemState`.
    - [x] Сквозной тест связи (PING/ACK) с STM32.
    - [x] Интеграция с `SystemState` для сохранения данных `0x03` и `0x04`.
    - [ ] Отправка команд из `Cmd_Srv`.
- [ ] **3.2. Network_Task (Core 0)**
    - [ ] Запуск WiFi (через Driver) и WebServer.
    - [ ] Реализация API эндпоинтов (через `Api_Srv`).
    - [ ] Регистрация GET `/api/status` для отображения данных обогревателя.
- [ ] **3.3. System_Task** (опционально)
    - [ ] Heartbeat LED, Watchdog, Heap Monitor.

## Фаза 4: Веб-интерфейс (Frontend)
*Цель: Пользовательский UI.*

- [ ] **4.1. API Layer**
    - [ ] `Api_Srv`: Генерация JSON status.
    - [ ] Обработка POST запросов управления.
- [ ] **4.2. SPA Client**
    - [ ] HTML/CSS/JS файлы.
    - [ ] Сборка и прошивка LittleFS.

## Фаза 5: Интеграция и Тесты
*Цель: Проверка стабильности.*

- [ ] **5.1. Stress Test**
    - [ ] Проверка на утечки памяти (Heap check).
    - [ ] Тест на восстановление связи (WiFi/UART).

## Фаза 6: Рефакторинг и Оптимизация (Advanced C++)
*Цель: Повышение уровня архитектуры до промышленного стандарта и оптимизация ресурсов.*

- [ ] **6.1. Абстракция каналов связи (Interfaces)**
    - [ ] Создание интерфейса `ICommChannel` (чисто виртуальный класс).
    - [ ] Перевод `Protocol_Srv` на работу с интерфейсом вместо жесткой привязки к UART.
- [ ] **6.2. Реактивная архитектура (Event Dispatcher)**
    - [ ] Внедрение шины событий (Event Bus) для связи между слоями.
    - [ ] Уход от циклического опроса `SystemState` к событийной модели.
- [ ] **6.3. Compile-Time Оптимизации**
    - [ ] Использование `constexpr` для расчетов CRC и конфигурационных констант.
- [ ] **6.4. Zero-Copy & Memory Management**
    - [ ] Оптимизация передачи фреймов через умные указатели или пулы памяти.
- [ ] **6.5. Система отказоустойчивости (Self-Healing)**
    - [ ] Реализация программного Watchdog на уровне отдельных задач FreeRTOS.
